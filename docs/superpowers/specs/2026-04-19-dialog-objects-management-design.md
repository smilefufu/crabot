# 对话对象管理设计

日期：2026-04-19

## 目标

将当前分散的 `熟人管理`、`待授权审批`、`会话管理` 能力收敛为一个新的统一页面 `对话对象管理`，围绕真正的管理对象重构信息架构与运行规则。

新页面需要同时满足：

- 用统一模型管理人与会话，不再把所有对象都压成狭义 session
- 让 UI 展示对象与底层实际参与处理的对象保持一致
- 将私聊权限、群聊权限、场景画像、完整记忆工作台挂到正确的主体上
- 将原 `/friends` 页面能力并入新页面
- 将 `/apply`、`/认主` 审批流并入新页面，但不与对象域混淆

## 命名与对象域

页面名：`对话对象管理`

页面中的核心对象域固定为三类：

1. `好友`
2. `私聊池`
3. `群聊`

`申请队列` 不作为第四个对象域存在，而是作为页面顶部的全局工作入口存在。

## 核心对象模型

### 好友

`好友` 是跨渠道的人类实体。

它承接：

- 基础资料
- 权限等级
- 渠道身份绑定（`channel_identities`）
- 统一私聊权限
- 统一私聊场景画像
- 统一私聊记忆工作台

一个好友可以绑定多个渠道身份。不同渠道的私聊，一旦归属到同一个好友，后续都归该好友统一管理。

### 私聊池

`私聊池` 是未归属私聊 session 的对象池。

条目粒度固定为：

- `channel_id + session_id`

它不是审批队列，也不是消息队列。它表示“当前还没有归属到任何好友的私聊 session 对象”。

私聊池条目可以长期存在，直到被归属到某个好友。

### 群聊

`群聊` 是独立的群 session 对象。

条目粒度固定为：

- `channel_id + session_id`

它承接：

- 群权限配置
- 群场景画像
- 群记忆工作台
- `master_in_group` 状态

群聊不进入好友体系。

## 权限与场景归属规则

### 私聊

已归属到好友的私聊：

- 不再拥有独立权限主体
- 不再拥有独立私聊场景主体
- 权限统一归好友
- 私聊场景统一归好友

因此，已归属私聊不会继续作为权限管理对象长期存在于会话管理语义中。

### 群聊

群聊始终是独立主体：

- 权限归群聊
- 场景归群聊
- 记忆工作台归群聊

运行时允许叠加发言人好友的只读上下文，但这种叠加不改变群聊对象本身的归属。

## 页面骨架

页面采用 `对象域 / 列表 / 详情工作区` 结构。

### 左侧：对象域导航

- 好友
- 私聊池
- 群聊

### 顶部：全局工具条

- 页面标题
- 全局搜索
- 当前筛选摘要
- 申请队列入口与数量 badge

### 中间：当前对象域列表

根据左侧所选对象域切换列表：

- `好友`：好友列表
- `私聊池`：未归属私聊 session 列表
- `群聊`：`master_in_group = true` 的群聊列表

### 右侧：详情工作区

根据当前选中对象切换详情模板。

## 各对象域的详情模板

### 好友详情

好友详情包含四个区块：

1. 基础信息
2. 渠道身份绑定
3. 统一私聊权限
4. 统一私聊场景与完整记忆工作台

原 `/friends` 页面承担的列表、详情、绑定、权限管理职责全部迁入这里。

### 私聊池详情

私聊池详情只承担归属决策相关职责，不承担长期权限与记忆管理。

区块包括：

1. 会话摘要
2. 来源渠道信息
3. 最近消息摘要
4. 归属动作

归属动作只有两类：

- `归到已有好友`
- `从私聊新建好友`

### 群聊详情

群聊详情包含：

1. 群基础信息
2. `master_in_group` 状态
3. 群权限配置
4. 群场景画像
5. 完整记忆工作台
6. 运行时上下文叠加说明

叠加说明需要明确表达：

- 群聊以群自己的场景为主
- 发言人的好友场景只以只读方式叠加

## 审批队列模式

`申请队列` 是事件流，不是对象域。

它通过顶部全局入口打开，进入一个临时审批模式：

- 中间区域切换为审批列表
- 右侧区域切换为审批详情与动作
- 关闭后返回之前所在对象域与选中对象

### 申请来源

审批事件仅来自：

- `/认主`
- `/apply`

审批队列与私聊池是不同概念：

- 私聊池管理未归属私聊 session 对象
- 审批队列管理主动触发的申请事件

两者可以互相引用，但不能合并建模。

### 审批列表分组

申请列表按意图分组：

1. `认主申请`
2. `普通申请`

组内按最新时间倒序。

### 认主申请动作

`/认主` 的语义不是“普通审批后再勾选 master”，而是天然的 master 归属流。

处理动作为：

- `并入现有 master`
- `创建 master`（仅当系统里当前没有 master）
- `拒绝`

### 普通申请动作

`/apply` 的处理动作为：

- `归到已有好友`
- `新建好友`
- `拒绝`

## 对象流转规则

### 私聊池到好友

新私聊进入系统后：

- 如果未归属任何好友，则进入私聊池

管理员可以执行：

- `归到已有好友`
- `从私聊新建好友`

归属完成后：

- 该私聊条目退出私聊池
- 后续由好友承接管理

### 好友解绑渠道身份

好友解绑某个渠道身份后：

- 该身份不再归属该好友
- 之后该渠道身份对应的私聊再次出现时，会重新进入私聊池

### 好友无渠道身份

当好友失去所有渠道身份时：

- 好友对象不删除
- 状态转为 `无渠道` / `失联`
- 默认不在主列表高频展示
- 仍保留权限、私聊场景、记忆和历史绑定记录
- 一旦重新绑定任意渠道身份，恢复为活跃好友

## 群聊准入规则

群聊只显示并处理 `master_in_group = true` 的群。

这是群级硬门槛，不是纯 UI 字段。

### 群聊显示规则

只有 `master_in_group = true` 的群聊出现在 `群聊` 域主列表里。

### 群消息处理规则

只有 `master_in_group = true` 的群聊消息才进入处理链。

否则：

- 群消息直接丢弃
- 群聊不在群聊域显示

### 状态变化后的处理

当 `master_in_group` 从 `true -> false`：

- 该群立即从列表隐藏
- 后续群消息停止进入处理链
- 群权限、群场景、群记忆保留

当 `master_in_group` 从 `false -> true`：

- 该群重新进入列表
- 复用原群对象
- 不新建群场景和群记忆

## master_in_group 状态同步机制

`master_in_group` 需要通过显式同步机制维护，而不能只靠手动编辑。

同步触发分三类：

1. 消息触发校验
2. 身份变更触发校验
3. 群成员变更触发校验

### 1. 消息触发校验

每次收到群消息时，都重新判断当前群中是否仍有 master。

这是最终兜底来源，因为它直接影响消息是否进入处理链。

### 2. 身份变更触发校验

以下情况需要重算相关群的 `master_in_group`：

- master 新增某渠道身份
- master 解绑某渠道身份
- `/认主` 审批通过后，master 新增某渠道身份
- 某好友权限变为 master
- 某 master 失去 master 权限

### 3. 群成员变更触发校验

如果 channel 能感知到以下变化，应立刻重算对应群状态：

- master 被踢出群
- master 退群
- 群成员列表变更导致群内已无 master

如果某个 channel 暂时拿不到这类事件，至少要靠下一条群消息触发的校验纠正状态。

## 运行时与 UI 一致性要求

新设计必须满足以下一致性：

### 对象一致性

UI 中可见对象必须和底层真实参与处理的对象一致。

### 流转一致性

私聊、好友、群聊之间的对象流转必须单向清晰，只有在显式解绑时才发生可逆流转。

### 权限一致性

- 已归属私聊：权限只归好友
- 群聊：权限只归群聊

### 记忆归属一致性

- 已归属私聊：场景与记忆只归好友
- 群聊：场景与记忆只归群聊
- 群聊叠加好友上下文时，不改变对象归属

## 页面迁移结果

完成后，以下旧页面能力应被新页面吸收：

- `/friends`
- 待授权审批页面
- 当前狭义 `会话管理` 页面中的可保留能力

其中：

- `好友` 域吸收原 `/friends`
- 顶部 `申请队列` 模式吸收审批页
- `私聊池 / 群聊` 吸收并重定义会话管理
